## GO-LIVE VERDICT: GO-WITH-REQUIRED-FIXES

The deployment artifacts are structurally sound: no secrets in the repo or image, the CORS env-gate works correctly, the Pages Function proxy streams SSE and blocks `_metrics` at the edge, cookie flags are production-safe by default, and the Anthropic cost surface is properly auth-gated. The `[no-secrets-in-image]`, `[secret-encrypted-at-rest]`, and `[server-side-gate-enforcement]` canon invariants are verifiably intact.

## Required before go-live

### HIGH-1: `/api/_metrics` is publicly accessible via the Railway direct URL (S3b)
**Owner:** Backend lane

The Pages Function correctly 404s `_metrics` at the public Pages origin. However, `GET /api/_metrics` on the Railway `*.up.railway.app` origin is unauthenticated and internet-reachable. Railway URLs are not secret — they appear in `API_ORIGIN` and are enumerable. The endpoint leaks operational diagnostics: per-ticker stage timings, vendor call latency, rate-limit headroom, 25 recent request traces with ticker symbols and parameters.

**Minimum fix:** Add a bearer-token gate on `GET /api/_metrics` in `main.py`. Read a new env var `METRICS_SECRET_TOKEN`; require it as `?token=<value>` or `Authorization: Bearer <value>`; return 401/403 without it. Owner sets the token in Railway Variables. The operator reads metrics via `https://<railway-url>/api/_metrics?token=<token>`. One env var, ~5 lines of code.

---

### HIGH-2: Anonymous, unthrottled, cost-bearing Massive vendor calls on `/api/ticker/*` and `/api/stream/*` (S5b)
**Owner:** Backend lane (code) + Owner (config / Cloudflare WAF)

There is no rate limit at any layer on the anonymous ticker endpoints. A cold miss on any ticker triggers 3 Massive vendor API calls. An arbitrary-ticker enumeration loop could exhaust the Massive rate limit and drive vendor cost against the owner's account. The in-memory 60s cache only helps for repeated requests on the same ticker.

**Minimum fix (one of):**
- **Preferred (owner-config):** Enable a Cloudflare Pages rate-limiting rule on `/api/ticker/*` and `/api/stream/*` (≤30 requests/minute per IP). Available on the Cloudflare free tier via "Rate Limiting" in the Pages project's associated zone, or via a `_headers` file with Cloudflare WAF rules.
- **Alternative (Backend):** Add a `TICKER_ALLOWLIST` env var (comma-separated, e.g., `TSLA,SPY,QQQ,AAPL,...`); 404 any ticker not in the list. This is the simplest server-side fix and requires only one env var in Railway Variables plus ~10 lines of code.
- **Acceptance (if owner accepts the risk):** Document explicitly that this is a low-traffic, private-link-only launch and that the vendor rate limit is the natural ceiling. This acceptance must be explicit and should be re-evaluated before any public promotion.

---

### HIGH-3: Stable-key absence with Postgres persistence causes silent data loss (S4e)
**Owner:** Owner (config) + Backend lane (advisory warning)

If `ACCOUNT_STORE=postgres` is set but `AUTH_SESSION_SIGNING_KEY` and `AI_KEY_ENCRYPTION_KEY` are absent, the app silently falls back to ephemeral per-process keys. With a persistent Postgres store, this means every redeploy: (a) logs all users out (session cookies become unverifiable) and (b) makes all stored encrypted AI keys permanently undecryptable. The current code logs only an INFO-level message — not visible as a problem.

**Fix (two parts):**
1. **Owner (required):** Generate stable random values for both keys and set them in Railway Variables before first deploy. The architecture contract §3.A mandates this; the runbook (§7 step 4) documents it. This is an ops action, not a code change.
2. **Backend lane (recommended):** Add a startup WARNING log (not a fatal exit) when `ACCOUNT_STORE=postgres` is detected and either key is absent. This makes the misconfiguration immediately visible in Railway's log stream. Example: `logger.warning("STABLE-KEY MISSING: ACCOUNT_STORE=postgres but AUTH_SESSION_SIGNING_KEY is unset. Users will be logged out on every restart. Set a stable key in Railway Variables.")`.
